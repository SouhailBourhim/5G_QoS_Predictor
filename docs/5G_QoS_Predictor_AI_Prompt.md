# 5G QoS Predictor — Project Specification for Implementation

## What You Are Building

An end-to-end machine learning system that predicts **SLA (Service Level Agreement) violations in 5G network slices** before they occur, providing 15 to 60 minutes of advance warning. This is a prototype of a **3GPP NWDAF (Network Data Analytics Function)** analytics module — specifically the `QoS Sustainability` analytics ID defined in 3GPP TS 23.288 §6.9.

The system covers: synthetic data generation → feature engineering → multi-horizon classification → FastAPI deployment → Streamlit dashboard → Docker containerization.

---

## Domain Context (Read Before Writing Any Code)

### 5G Network Slices

There are 3 slice types, each with different SLA thresholds:

| Slice | Use Case | Key SLA Requirements |
|-------|----------|----------------------|
| **eMBB** (enhanced Mobile Broadband) | Video streaming, web browsing | DL throughput ≥ 50 Mbps, latency ≤ 30 ms, packet loss ≤ 1% |
| **URLLC** (Ultra-Reliable Low-Latency) | Factory robots, remote surgery | Latency ≤ 5 ms, reliability ≥ 99.999%, jitter ≤ 1 ms |
| **mMTC** (massive Machine-Type Comms) | IoT sensors, smart meters | Message delivery ≥ 95%, latency ≤ 1000 ms, density ≥ 10⁶ devices/km² |

### Why This Matters

Current operators use **static threshold monitoring**: an alarm fires *after* a KPI crosses a limit. This is reactive — the SLA is already breached before any action can be taken. This system predicts violations *before* they happen, giving operators 15–60 minutes to:
- Reallocate Physical Resource Blocks (PRBs)
- Activate traffic shaping or load balancing
- Preempt lower-priority slice resources

## Phase 2 — Synthetic Slice Data Generator

**Output:** ~77,760 rows total (25,920 per slice), stored as Parquet.

### Layer 1: Daily Traffic Patterns

Each slice type has a unique intraday load profile:

```python
import numpy as np

def embb_pattern(t):
    """Triple-hump: morning commute, lunch, evening streaming."""
    return (0.15 * np.exp(-((t - 8.5)**2) / (2 * 1.8**2)) +
            0.30 * np.exp(-((t - 12.5)**2) / (2 * 1.5**2)) +
            0.45 * np.exp(-((t - 20.5)**2) / (2 * 2.2**2)) +
            0.08)

def urllc_pattern(t):
    """Sigmoid business-hours plateau (7:30 AM to 5:30 PM)."""
    sigmoid = lambda x: 1 / (1 + np.exp(-x))
    return sigmoid(3 * (t - 7.5)) - sigmoid(3 * (t - 17.5))

def mmtc_pattern(t, delta=1.0, epsilon=0.1):
    """Periodic impulse bursts from sensor reporting cycles."""
    return 0.15 + 0.7 * (np.mod(t, delta) < epsilon).astype(float)
```

### Layer 3: KPI Physical Relationships

KPIs are derived from the load level using physical relationships:

```python
# For KPIs that INCREASE with load (latency, jitter, packet_loss, prb_util, active_users):
```

|-----|---------------------|------------------------|
| DL Throughput | Inverse | Log-normal |
| Latency | Proportional | Gamma |
| Jitter | Proportional | Exponential |
| Packet Loss | Proportional | Beta [0, 100] |
| PRB Utilization | Proportional | Normal |
| Active Users | Proportional | Normal |
| Reliability | Inverse | Normal (narrow) |

### Layer 5: Violation Event Injection

Six event types with known ground truth labels:

| Event Type | Affected Slices | Onset | Buildup Duration |
|------------|----------------|-------|-----------------|
| Traffic Surge | eMBB | Sudden | 5 min |
| Gradual Congestion | eMBB, mMTC | Slow | 30–120 min |
| Interference | URLLC, eMBB | Intermittent | 0 min |
| Hardware Degradation | All | Very slow | 60–480 min |
| Resource Starvation | URLLC | Correlated | 15–60 min |
| IoT Storm | mMTC | Spike | 0 min |

The degradation during buildup follows a linear ramp:

```python
def compute_degradation(t, t_buildup_start, t_event_start, t_event_end, severity):
    if t_buildup_start <= t < t_event_start:
        progress = (t - t_buildup_start) / (t_event_start - t_buildup_start)
        return severity * 0.7 * progress
    elif t_event_start <= t <= t_event_end:
        return severity * np.random.normal(1.0, 0.1)
    return 0.0
```

**Important:** Severity factor `s` is drawn from `Uniform(0.3, 1.0)` per event. The degradation is added to the KPI in the direction that pushes it toward SLA violation.

```python
def build_targets(df, sla_thresholds, horizons=[15, 30, 60]):
    # 1. Instantaneous SLA breach (boolean per KPI)
```

### Category 6: SLA Proximity Features (30 features) ⭐ MOST IMPORTANT

This is the highest-impact feature category. Do not skip or simplify.

```python
def compute_sla_proximity(df, kpi, threshold, direction, smooth_window=3):
    """
    direction: 'min' means KPI must stay ABOVE threshold (e.g., throughput)
               'max' means KPI must stay BELOW threshold (e.g., latency)
    """
    if direction == 'min':
        margin = df[kpi] - threshold        # Positive = safe
    else:
        margin = threshold - df[kpi]        # Positive = safe

    df[f'{kpi}_sla_margin'] = margin
    df[f'{kpi}_sla_margin_norm'] = margin / (abs(threshold) + 1e-9)

    # Rolling minimum margin (worst case in recent window)
    for w in [6, 12, 36]:
        df[f'{kpi}_sla_margin_min_{w}'] = margin.rolling(w, min_periods=1).min()

    # Smoothed derivative for time-to-breach estimation
    smooth_diff = df[kpi].diff(1).rolling(smooth_window, min_periods=1).mean()
    if direction == 'min':
        approaching = smooth_diff < 0       # KPI decreasing toward min threshold
        rate = -smooth_diff
    else:
        approaching = smooth_diff > 0       # KPI increasing toward max threshold
        rate = smooth_diff

    # Time-to-breach: minutes until SLA crossed at current rate
    # 999 = not approaching, or safe for >999 minutes
    time_to_breach = np.where(
        approaching & (rate > 1e-9),
        np.abs(margin) / rate * 5,          # * 5 because each step = 5 minutes
        999.0
    )
    df[f'{kpi}_time_to_breach'] = np.clip(time_to_breach, 0, 999)

    return df
```

## Phase 4 — Data Splitting

**Critical rule: never use random splits on time-series data.**

```python
# 90 days total, 5-minute granularity
# Train: Days 1–60   (25,920 × 3 rows × 0.67 = ~51,840 rows per slice)
# Val:   Days 61–75  (used ONLY for threshold optimization)
# Test:  Days 76–90  (used ONLY for final reported metrics)

def temporal_split(df):
    df = df.sort_values('timestamp')
    cutoff_train = df['timestamp'].min() + pd.Timedelta(days=60)
    cutoff_val   = df['timestamp'].min() + pd.Timedelta(days=75)

    train = df[df['timestamp'] <  cutoff_train]
    val   = df[(df['timestamp'] >= cutoff_train) & (df['timestamp'] < cutoff_val)]
    test  = df[df['timestamp'] >= cutoff_val]

    # ALWAYS verify no leakage
    assert train['timestamp'].max() < val['timestamp'].min()
    assert val['timestamp'].max()   < test['timestamp'].min()

    return train, val, test
```

**Threshold Optimization (Critical Step):**

```python
from sklearn.metrics import precision_recall_curve

def find_optimal_threshold(clf, X_val, y_val, min_recall=0.90):
    """
    Find the highest-precision threshold that still achieves min_recall.
    Run this on VALIDATION set only. Never on test set.
    """
    y_proba = clf.predict_proba(X_val)[:, 1]
    precisions, recalls, thresholds = precision_recall_curve(y_val, y_proba)

    # Filter to only thresholds achieving min_recall
    valid_mask = recalls[:-1] >= min_recall
    if not valid_mask.any():
        return 0.5  # fallback

    # Among valid thresholds, pick the one with highest precision
    best_idx = np.argmax(precisions[:-1][valid_mask])
    return thresholds[valid_mask][best_idx]

# Usage:
optimal_threshold = find_optimal_threshold(clf, X_val, y_val, min_recall=0.90)
y_pred = (clf.predict_proba(X_test)[:, 1] >= optimal_threshold).astype(int)
```

```python
def generate_recommendations(top_shap_features, slice_type, probabilities):

    feature_names = [f['feature'] for f in top_shap_features]
    if any('prb_util' in f for f in feature_names):
        recs.append(RECOMMENDATIONS['prb_util_high'])
    if any('cross_slice' in f or 'embb_prb' in f for f in feature_names):
```

## Key Constraints and Pitfalls

- **No random train/test splits** — always use strict temporal partitioning
- **Threshold optimization on validation set only** — never touch test set until final evaluation
- **Use AUC-PR as the primary metric**, not AUC-ROC, due to class imbalance
- **Tag every generated row with its event type** — required for per-event recall analysis in Pillar 4
- **SHAP must use `TreeExplainer`** for XGBoost/LightGBM, not the generic `Explainer`
- **Store calibration params as JSON** — do not hardcode in the generator
- **Parquet format for all datasets** — not CSV, for efficient time-series I/O
- **All feature computation uses `shift()` and `rolling()`** — never `iloc` with future indices
